## 1. What is Version Control?

Version control systems (VCS) track changes to code over time, allowing you to:

- Revert to previous versions of your project.
- Collaborate without overwriting others‚Äô work.
- Experiment safely with new features (using branches).

**Types of VCS:**

- **Centralized (e.g., SVN):** A single server stores all versions.
- **Distributed (e.g., Git):** Every developer has a full copy of the project history.

**Why Git?**

- **Speed:** Git operates locally, so most actions are instantaneous.
- **Flexibility:** Branching, merging, and stashing are seamless.
- **Security:** Data integrity is ensured via cryptographic hashing.

## 4. Git Basics: Core Concepts

**Repositories (Repos)**

A folder where Git tracks changes. It includes:

- **Working Directory:** Your current files.
- **Staging Area:** Files ready to be committed.
- **Commit History:** Snapshots of your project over time.

**Branches**

- **Main/Master:** The default branch (keep it stable).
- **Feature Branches:** Isolate new work (e.g., login-feature).

**Commits**

A commit is a saved snapshot of your code. Always include a meaningful message:

```bash
git commit -m "Fix user login validation error"  # Good  
git commit -m "Update code"                      # Avoid  
```

**Remote vs. Local**

- **Local:** The repo on your machine.
- **Remote:** The repo hosted on GitHub (e.g., origin).

## 5. Daily Use Git Commands

**Setup & Cloning**

```bash
git init                          # Start a new repo  
git clone https://github.com/user/repo.git  # Copy a repo to your machine  
```

**Saving Changes**

```bash
git add index.html               # Stage one file  
git add .                        # Stage all changes  
git commit -m "Update homepage"  # Save staged changes  
```

**Viewing History**

```bash
git log                          # Show commit history  
git log --oneline --graph        # Compact view with branch structure  
git diff                         # Compare unstaged changes  
```

**Branching**

```bash
git branch                       # List all branches  
git checkout -b login-feature    # Create and switch to a new branch  
git checkout main                # Switch back to main  
git merge login-feature          # Merge login-feature into main  
```

**Syncing with GitHub**

```bash
git pull origin main             # Fetch changes from GitHub and merge  
git push origin login-feature    # Upload your branch to GitHub  
git remote -v                    # Check linked remotes  
```

**Undoing Mistakes**

```bash
git reset HEAD~1                 # Undo the last commit (keeps changes)  
git checkout -- file.txt         # Discard unstaged changes in a file  
git stash                        # Save changes temporarily  
git stash pop                    # Restore stashed changes  
```

## 6. GitHub Workflows

**Collaborating via Pull Requests**

1. Fork a repo on GitHub.
2. Clone your fork locally.
3. Create a branch, make changes, and push it.
4. Open a Pull Request to propose merging your changes into the original repo.
5. Discuss, refine, and merge!

**Resolving Merge Conflicts**

1. Run `git pull` to fetch the latest changes.
2. Open the conflicted file and look for `<<<<<<< HEAD` and `>>>>>>>` markers.
3. Edit the file to keep the correct code.
4. Stage, commit, and push the resolved file.

## 7. Pro Tips for Avoiding Pitfalls

- **Commit Small Changes:** Break work into logical chunks (e.g., ‚ÄúAdd button CSS‚Äù vs. ‚ÄúUpdate entire UI‚Äù).
- **Write Descriptive Messages:** Explain what changed and why.
- **Use `.gitignore`:** Exclude files like `node_modules`, `.env`, or logs.
- **Pull Before Pushing:** Always sync with the remote to avoid conflicts.
- **Learn Git Aliases:** Speed up commands (e.g., `git config --global alias.co checkout`).
